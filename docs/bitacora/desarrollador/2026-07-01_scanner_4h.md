# Bitácora Desarrollador — 2026-07-01: Scanner 4H multi-activo

## Qué se implementó

Reemplazo completo de la estrategia base (cruce EMA21/EMA50 en 1H) por un scanner
4H que evalúa múltiples activos (cripto + acciones) buscando setups de pullback
a la EMA20 en sobreventa, dentro de tendencias alcistas de largo plazo.

### Archivos nuevos

| Archivo | Rol |
|---|---|
| `src/scanner.py` | Lógica del scanner: condición 4H, pre-filtros 1D/1W, llamada al cerebro |
| `src/watchlist.py` | Lista editable de activos a escanear (cripto + acciones) |
| `scripts/run_scanner.py` | Entrypoint: loop cada 4H con `--ahora` para escaneo inmediato |
| `scripts/consultar_cerebro.py` | Consulta manual ad-hoc al cerebro sobre una señal dada |

### Archivos modificados

| Archivo | Qué cambió |
|---|---|
| `src/brain.py` | `contextualizar()` nueva función; `_validar_contexto` acepta tickers sin "/" |
| `src/notifier.py` | `notificar_scanner()`: un solo mensaje combinando alerta + análisis |
| `requirements.txt` | `yfinance>=0.2.0` agregado para datos de acciones |

## Estrategia

### Condición de entrada (pre-filtro mecánico — sin LLM)

Evaluada en orden, descartando el activo en cuanto falla alguna:

1. **4H:** `abs(precio - EMA20) / EMA20 ≤ 2%` **y** `RSI(14) < 35`
2. **1D:** `close > EMA50` (tendencia alcista diaria)
3. **1W:** `close > EMA20` (tendencia alcista semanal)

Si pasa los 3 filtros → se notifica a Telegram y se llama al cerebro.

### Rol del cerebro (`contextualizar()`)

El cerebro **no valida ni veta** la señal. La estrategia ya está confirmada
mecánicamente. El cerebro aporta contexto cualitativo:

- Calidad y madurez de la tendencia
- Estructura de precio: niveles clave cercanos, soporte/resistencia relevante
- Comportamiento del volumen
- Factores de riesgo concretos
- Lectura general del setup (fuerte / moderado / con reservas)

Devuelve: `analisis` (5-6 líneas en prosa), `nivel_atencion` (alto/medio/bajo),
`alertas` (máx 3 factores de riesgo).

Esto es distinto a `analizar()` (que sí puede vetar y se usa en el scheduler 1H).

## Decisiones de diseño

### Pre-filtro antes del cerebro
El pre-filtro reduce el número de llamadas LLM. En cada vela 4H, la mayoría
de los activos no cumplen la condición → cero llamadas al LLM. Solo los activos
que pasan los 3 filtros gastan API.

### Separación brain.py: analizar() vs contextualizar()
Dos funciones públicas con roles distintos:
- `analizar()`: usado por el scheduler 1H. Puede confirmar, vetar o dar neutral.
  Tiene schema estricto con `multiplicador_riesgo`.
- `contextualizar()`: usado por el scanner. Solo produce texto de contexto.
  Fallback silencioso si el LLM falla — la alerta ya fue enviada igual.

### Un solo mensaje a Telegram
La alerta mecánica y el análisis del cerebro se combinan en un único mensaje
(`notificar_scanner()`). El usuario ve todo el contexto en una sola notificación.

### Soporte de acciones (yfinance)
Detección automática por formato: tickers con `/` → ccxt (cripto), sin `/` → yfinance (acciones).
Para 4H en acciones: descarga 1H y resamplea a 4H con `dropna(subset=["close"])`
para eliminar huecos de mercado cerrado.

### Jerarquía 1W en el cerebro
`_SYSTEM_PROMPT` actualizado para incluir `1W` como timeframe de máxima prioridad.
Si el contexto incluye datos `1w` (siempre para el scanner), LONG con 1W bajista
se veta siempre. Esto refuerza la regla que ya aplica el pre-filtro mecánico.

### par sin "/" en brain.py
`_validar_contexto` ya no requiere "/" en el campo `par`, para aceptar tickers
de acciones (ej. "AAPL") además de pares cripto (ej. "BTC/USDT").

## Uso

```bash
# Rebuild (necesario después de agregar yfinance)
docker compose build

# Escaneo inmediato (sin esperar cierre de vela)
docker compose run --rm brain python scripts/run_scanner.py --ahora

# Loop automático cada cierre de vela 4H
docker compose run --rm brain python scripts/run_scanner.py

# Consulta manual puntual
docker compose run --rm brain python scripts/consultar_cerebro.py --senal LONG --par BTC/USDT
```
