## Arquitectura implementada en `brain.py`

### Flujo de `analizar(contexto)`

1. **Validación de entrada** (`_validar_contexto` + `_validar_timeframe`)
   - Verifica todos los campos obligatorios: `par`, `timestamp`, `mercado_tipo`, `senal_base`, `portfolio`, `timeframes`
   - Verifica formato de `par` (contiene "/"), `timestamp` (parseable ISO 8601), enums (`mercado_tipo`, `senal_base`, `tendencia`)
   - Verifica rangos: `riesgo_disponible_pct` en [0,1], `rsi` en [0,100], `atr > 0`, `volumen >= 0`, `volumen_promedio > 0`
   - Verifica `maximos_recientes` y `minimos_recientes`: listas de exactamente 5 elementos
   - Si falla cualquier validación → fallback inmediato con `ENTRADA_INVALIDA: <razón>`, sin llamar al LLM

2. **Formateo del mensaje de usuario**
   - Prefijo: `"Evalúa la siguiente señal de trading:"`
   - Contexto completo serializado con `json.dumps(indent=2, ensure_ascii=False)`

3. **Llamada al LLM**
   - `model` desde `os.environ.get("ANTHROPIC_MODEL", "claude-sonnet-4-6")`
   - `max_tokens=1024`, `temperature=0.1`
   - `tool_choice={"type": "tool", "name": "decision_cerebro"}` — fuerza tool use
   - Captura `anthropic.APIError` y cualquier `Exception` → fallback `API_ERROR`

4. **Verificación del tool use**
   - Itera sobre `response.content` buscando bloque `tool_use` con `name="decision_cerebro"`
   - Si no encontrado → fallback `TOOL_NO_LLAMADO`

5. **Validación y corrección de salida** (`_validar_y_corregir_salida`)
   - Enums inválidos (`regimen`, `evaluacion_senal`) → `return None` → fallback `SCHEMA_INVALIDO`
   - Floats fuera de rango (`confianza_regimen`, `conviccion`, `multiplicador_riesgo`) → clampear + loguear anomalía + agregar a `alertas`
   - `multiplicador_riesgo > 1.0` → clampear a 1.0, loguear como anomalía
   - `factores_clave` vacío o no es lista → `return None` → fallback `SCHEMA_INVALIDO`
   - `racional` vacío → `return None` → fallback `SCHEMA_INVALIDO`
   - `racional` > 280 chars → truncar a 277 + `"..."`, loguear anomalía
   - `alertas` no lista → no falla (se usa lista vacía)

6. **Caso borde `riesgo_disponible_pct == 0.0`**
   - Aplicado post-validación en `_validar_y_corregir_salida`
   - Si `riesgo_disponible_pct == 0.0` y `multiplicador_riesgo != 0.0` → forzar a 0.0, loguear, agregar a `alertas`

## Decisiones de implementación

### `_fallback_con_razon(razon: str) -> DecisionCerebro`
Función helper que loguea la razón y devuelve el fallback estándar con `alertas = ["FALLBACK_ACTIVADO", razon]`. Esto permite distinguir la causa en el log de SQLite (Fase 3).

### Captura de excepciones en la llamada al LLM
Se captura `anthropic.APIError` (errores HTTP de la API: 401, 429, 500, etc.) y luego `Exception` genérica como red de seguridad adicional. Ambas → `API_ERROR`. Nunca se propaga excepción al caller.

### `_validar_y_corregir_salida` devuelve `None` en fallos irrecuperables
En lugar de devolver directamente el fallback, devuelve `None` para que `analizar()` llame a `_fallback_con_razon("SCHEMA_INVALIDO")` con logging consistente.

### `DecisionCerebro(**resultado)` para el fallback
El fallback usa `dict(_FALLBACK)` como copia mutable para poder sobrescribir `alertas` con la razón específica antes de construir el TypedDict.

### El caso borde `senal_base = "NONE"` es responsabilidad del LLM
El system prompt lo instruye explícitamente. El código no fuerza `evaluacion_senal = "neutral"` post-recepción, conforme a `docs/contrato_cerebro.md` — "Solo el system prompt lo instruye. El código no lo fuerza."
